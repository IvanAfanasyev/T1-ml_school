# Что сделано сверх ТЗ

Документ фиксирует функции, которые выходят за рамки базового MVP из технического задания и усиливают демонстрационный проект.

## 1. Диалоговый агент

По ТЗ достаточно было показать сценарий, где пользователь описывает задачу, а система возвращает несколько подходящих услуг. В проекте дополнительно реализован диалоговый режим:

- агент уточняет недостающие параметры;
- понимает ответы вроде `любой бюджет`, `мне все равно`, `в Москве`;
- хранит контекст одного чата;
- отделяет рабочие запросы от нерелевантного общения;
- защищает основной сценарий от prompt injection.

## 2. Top-3 провайдера вместо простого top-3 сервисов

В базовом сценарии можно было ранжировать отдельные сервисы. В проекте обычный запрос возвращает top-3 провайдера, где каждый провайдер представлен наиболее релевантным сервисом. Это ближе к задаче выбора облачного поставщика, а не отдельной тарифной строки.

## 3. Комплексные инфраструктурные связки

Для сложных запросов система выделяет несколько ролей: backend, база данных, хранилище, backup, балансировщик и другие компоненты. Затем она собирает связку сервисов от одного провайдера. Например, для интернет-магазина система может подобрать compute, managed PostgreSQL, Object Storage, Backup и Load Balancer как единый вариант.

## 4. Гибридный поиск

Помимо правил и тегов используется гибридный retrieval:

- embeddings для смысловой близости;
- BM25 для текстовых совпадений;
- структурированное совпадение по технологиям, сценариям, компонентам, региону, 152-ФЗ и бюджету.

Это дает более устойчивый подбор, чем простой поиск по словам.

## 5. LLM-извлечение структуры запроса

LLM используется не для принятия финального решения, а для извлечения структуры пользовательского текста:

- тип запроса: одиночный сервис или связка;
- технологии;
- регион;
- бюджет;
- нужные компоненты;
- дополнительные требования.

Ранжирование и фильтрация остаются в коде.

## 6. Нормализация разговорных формулировок

Система учитывает русские и разговорные варианты:

- `постгрес`, `посгрискл`, `postgres` -> `postgresql`;
- `майскл`, `мускул` -> `mysql`;
- `кубер`, `k8s` -> `kubernetes`;
- `любой бюджет`, `мне все равно` -> параметр не ограничивает поиск.

## 7. Проверка бюджета и тарифных позиций

Добавлена логика, которая не берет первую дешевую тарифную строку как цену всей услуги. Например, для Kubernetes цена за ГБ диска не считается полной стоимостью кластера. Если пользователь просит сервер за 1 рубль, система не должна выдавать это как подходящий вариант, а должна показать ближайшие реальные варианты и отметить бюджетное ограничение.

## 8. Frontend с чатовой историей

Помимо backend API сделан статический frontend:

- витрина сервисов;
- карточки услуг и тарифов;
- диалоговый агент;
- история чата в `localStorage`;
- сброс истории чата;
- страница о проекте.

## 9. Деплойная схема

Проект подготовлен к запуску в Docker и на виртуальной машине:

- backend запускается в Docker-контейнере;
- frontend отдается как статические файлы;
- Nginx проксирует `/api`, `/health`, `/docs` и `/openapi.json` на backend;
- документация описывает локальный запуск, тестирование и деплой.

## 10. Автоматические тесты

Добавлены тесты для ключевых сценариев:

- API;
- диалоговых уточнений;
- prompt injection;
- обработки бюджета;
- выбора тарифных позиций;
- регионов;
- scoring;
- комплексных связок.

Текущая проверка:

```bash
python3 -m unittest
```
