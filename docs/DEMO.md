# Demo-сценарий защиты

## Цель

За 3–5 минут показать:

- диалоговый режим;
- поиск ближайшего региона;
- подбор инфраструктурной связки;
- объяснимое ранжирование.

## 1. Диалог

Запрос:

```text
Нужно подобрать облако для веб-приложения.
```

Ожидаемо: агент уточняет стек, регион, бюджет, 152-ФЗ, БД/storage.

Ответ:

```text
Backend на Python, нужна PostgreSQL-база и object storage. Данные персональные, Москва, бюджет до 200000 рублей.
```

Что сказать:

```text
Если пользователь дал неполный запрос, агент не выдаёт случайный результат, а уточняет недостающие параметры.
```

## 2. Ближайший регион

Запрос:

```text
ВМ с 152-ФЗ во Владивостоке, до 10000 рублей.
```

Что показать:

- система поняла compute;
- применила 152-ФЗ;
- нашла ближайший доступный регион;
- объяснила замену региона.

## 3. Инфраструктурная связка

Запрос:

```text
Нужно развернуть веб-сервис для обработки персональных данных. Backend на Python, нужна виртуальная машина, управляемая PostgreSQL-база и объектное хранилище для бэкапов на 500 ГБ. Регион Москва, 152-ФЗ обязателен, бюджет до 200000 рублей.
```

Что показать:

- compute;
- managed DB;
- object storage;
- подбор компонентов у провайдера;
- объяснение совпавших критериев.

## 4. Ранжирование

Короткая формулировка:

```text
Система извлекает структуру запроса, применяет фильтры, затем ищет top-30 кандидатов через BM25 и embeddings. После этого считает entity_match_score по компонентам, стеку, use case, бюджету и требованиям.
```

Формулы:

```text
retrieval_score = 0.7 * embedding_score + 0.3 * bm25_score
final_score = 0.7 * retrieval_score + 0.3 * entity_match_score
```

## Что лучше не показывать первым

Для основного демо не начинать с очень спорных edge cases. Их лучше оставить для вопросов.
